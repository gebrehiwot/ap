Preparing for Advanced Programming - Beyond the Basics

1. Encapsulation: State Management & Data Hiding

At its core, Encapsulation is about **bundling data (attributes) and methods (behaviors) that operate on the data within a single unit**, and restricting direct access to some of the component's internals.

Beyond Private Variables: Protecting Invariants

Teaching Point: Emphasize that private is just the syntax. The principle is about **maintaining object integrity** and preventing illegal states. Draw a box around the `balance` and `deposit` method, showing how they're a unit. 
public class BankAccount {
    private double balance; // Crucial state, should not be directly modifiable

    public BankAccount(double initialBalance) {
        if (initialBalance < 0) {
            throw new IllegalArgumentException("Initial balance cannot be negative.");
        }
        this.balance = initialBalance;
    }

    // Controlled modification
    public void deposit(double amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("Deposit amount must be positive.");
        }
        this.balance += amount;
    }

    // Read-only access
    public double getBalance() {
        return balance;
    }

    // Bad practice: Exposes internal mutable state, breaking encapsulation
    // public Date getCreationDate() { return this.creationDate; }
    // Good practice: Defensive Copying
    // public Date getCreationDate() { return new Date(this.creationDate.getTime()); }
}
Advanced Concept: Immutability. For critical data, consider making objects immutable (all fields final, no setters). This inherently guarantees thread safety and simplifies reasoning about state. Example: Java's String class.

2. Inheritance: Specialization & Code Reuse

Inheritance allows a class (subclass/child) to acquire the fields and methods of another class (superclass/parent). It models an **"is-a"** relationship, promoting code reuse and forming a hierarchical structure.

Composition over Inheritance (Advanced Design Principle)

Teaching Point: Explain why "composition over inheritance" is often preferred in advanced design. Draw two diagrams: one showing a deep inheritance hierarchy, and another showing an object "having" capabilities. 
// Inheritance (is-a): Not always ideal for behavior reuse
class Animal { /* ... common animal stuff ... */ }
class Dog extends Animal { /* ... dog specific stuff ... */ }

// Composition (has-a): More flexible
interface Flyable { void fly(); }
class Bird implements Flyable {
    private Flyable wingMechanism; // Bird "has a" flying mechanism
    public Bird(Flyable mechanism) { this.wingMechanism = mechanism; }
    public void fly() { wingMechanism.fly(); }
}
Advanced Concept: Liskov Substitution Principle (LSP). A subclass should be substitutable for its superclass without altering the correctness of the program. If you have a List<Animal>, replacing an Animal with a Dog should not break the code.

3. Abstraction: Complexity Hiding & Contract Definition

Abstraction focuses on hiding the complex implementation details and showing only the necessary features to the user. In Java, this is primarily achieved using **abstract classes** and **interfaces**.

Abstract Classes vs. Interfaces: When to Choose Which?

Teaching Point: Draw a Venn diagram or a table contrasting the two. Abstract classes for "base identity with shared state/methods," interfaces for "pure behavior contracts." 
// Abstract Class: Partial implementation, shared state possible
abstract class PaymentProcessor {
    protected String transactionId; // Can have state
    public PaymentProcessor(String id) { this.transactionId = id; }
    public abstract void processPayment(double amount); // Abstract method
    public void logTransaction() { // Concrete method
        System.out.println("Logging transaction: " + transactionId);
    }
}

// Interface: Pure contract, no state
interface Authenticatable {
    boolean authenticate(String username, String password); // Only method signature
    default void resetAttempts() { // Default method in Java 8+
        System.out.println("Authentication attempts reset.");
    }
}
Advanced Concept: Functional Interfaces. Introduced in Java 8, these are interfaces with a single abstract method, enabling Lambda Expressions. Essential for modern Java development (e.g., Runnable, Comparator).

4. Polymorphism: Dynamic Dispatch & Flexible Code

Meaning "many forms," Polymorphism allows objects of different classes to be treated as objects of a common superclass or interface. It enables a single interface to represent various underlying forms, primarily through **Method Overriding** (runtime) and **Method Overloading** (compile-time).

Runtime vs. Compile-time Polymorphism

Teaching Point: Use the "Animal speaks" analogy. Draw an `Animal` reference pointing to different objects (`Dog`, `Cat`) and show how `makeSound()` behaves differently. 
class Animal {
    public void makeSound() { System.out.println("Animal makes a sound."); }
}

class Dog extends Animal {
    @Override // Runtime Polymorphism (Overriding)
    public void makeSound() { System.out.println("Dog barks!"); }
    
    // Compile-time Polymorphism (Overloading)
    public void makeSound(String mood) {
        System.out.println("Dog barks when " + mood);
    }
}

public class Zoo {
    public static void main(String[] args) {
        Animal myAnimal = new Dog(); // Polymorphism in action!
        myAnimal.makeSound(); // Calls Dog's makeSound() - dynamic dispatch
        
        Dog myDog = new Dog();
        myDog.makeSound("happy"); // Calls overloaded method
    }
}
Advanced Concept: Covariant Return Types. A method overriding another can return a more specific type than the overridden method, as long as it's a subtype of the original. This enhances flexibility.
