PILLAR 01

Encapsulation

🔒 data hiding · integrity

private fields, public methods.
Guard internal state from invalid changes.
→ setter validation, defensive copies 
public class BankAccount {
    private long balance;        // hidden

    public void deposit(long amount) {
        if (amount > 0) {
            this.balance += amount;
        } // else: reject, log, exception
    }

    public long getBalance() {
        return balance;
    }
}
✏️ draw the "wall" around balance — only methods can touch it
PILLAR 02

Abstraction

interface PaymentProcessor {
    void pay(BigDecimal amount);
}

class StripeProcessor implements PaymentProcessor {
    @Override
    public void pay(BigDecimal amount) {
        // complex REST calls hidden
        System.out.println("💰 Stripe: " + amount);
    }
}

📡 hide complexity, expose contract

• interface / abstract class
• client only knows what, not how
• reduces coupling, increases modularity
✏️ label: "interface = contract"
PILLAR 03

Polymorphism

🌀 one interface · many implementations

// Runtime polymorphism
List<Notification> channels = List.of(
    new EmailNotification(),
    new SmsNotification()
);

for (Notification n : channels) {
    n.send("Your code is 4421");   // which send()?
}   // JVM chooses based on actual object (vtable)
✔ method overriding
✔ parent reference, child object
✔ enables plug‑in architectures
✔ open/closed principle
ARCHITECTURE DEEP

Inheritance · Composition

// inheritance (is-a)
class ElectricCar extends Car { }
// composition (has-a)
class ElectricCar {
    private Engine engine;   // injected
    private Battery battery;
}

⛓️ favor composition over inheritance — more flexible, less fragile.

Stack · Heap

📚 Stack (per‑thread)

  • primitives (int, boolean)
  • object references
  • method calls / frames

🗑️ Heap (shared)

  • new objects live here
  • garbage collection
  • arrays, instance fields

    int x = 5;             // stack
    Account acc = new Account(); // ref (stack) → object (heap)
